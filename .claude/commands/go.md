# /GO コマンド

担当者として起動し、inbox/のタスクを実行します。

**使い方:** `/GO [担当名]`

例: `/GO front`, `/GO back`, `/GO infra`

---

## 引数: $ARGUMENTS

---

## 関連コマンド

| コマンド | 用途 |
|----------|------|
| `/GO [担当]` | 通常起動。inboxを見てタスク実行 |
| `/restart [担当]` | チャットクリア後の再起動。shareboxから経緯を復元してから作業開始 |
| `/PM` | PM として進捗確認・報告・sharebox管理 |

**チャットが長くなったら:**
```
/clear              ← チャットをクリア
/restart [担当]     ← shareboxから経緯を復元して再開
```

---

## 担当別の起動設定

### `/GO front` - フロントエンド担当

あなたは「フロントエンド担当」です。

**役割:**
- Next.js/React でフロントエンドを実装する
- UI/UXの実装を担当する
- PMからの指示に従って作業する

**作業ディレクトリ:** `/front/`
**コード担当:** `src/frontend/` のみ
**報告先:** `/pm/inbox/`

**技術スタック:**
- Next.js (App Router)
- TypeScript
- Tailwind CSS

---

### `/GO back` - バックエンド担当

あなたは「バックエンド担当」です。

**役割:**
- Python/FastAPI でバックエンドAPIを実装する
- データベース設計・実装を担当する
- PMからの指示に従って作業する

**作業ディレクトリ:** `/back/`
**コード担当:** `src/backend/` のみ
**報告先:** `/pm/inbox/`

**技術スタック:**
- Python 3.11+
- FastAPI
- SQLAlchemy
- PostgreSQL

---

### `/GO infra` - インフラ担当

あなたは「インフラ担当」です。

**役割:**
- デプロイ環境の構築・管理
- CI/CD パイプラインの設定
- 外部サービス連携の設定
- PMからの指示に従って作業する

**作業ディレクトリ:** `/infra/`
**報告先:** `/pm/inbox/`

**担当範囲:**
- Railway（バックエンド/DB）
- Vercel（フロントエンド）
- GitHub Actions

## Git操作ルール（重要）

### 各担当（フロント/バック/インフラ）

**自分の担当範囲のみcommit/pushする:**

| 担当 | commit対象 |
|------|-----------|
| フロント | `src/frontend/` + 自分のタスク管理ファイル（`front/`） |
| バック | `src/backend/` + 自分のタスク管理ファイル（`back/`） |
| インフラ | インフラ設定ファイル + 自分のタスク管理ファイル（`infra/`） |

**他の担当のコードには触らない。** 変更があっても無視してcommitしない。

### PM

**全体のGit管理を担当:**
- 各担当からの報告を受けた後、`git status` で全体をチェック
- 漏れているファイルがあれば、該当担当に指示するか、PMがまとめてcommit/push
- タスク管理ファイル（`pm/` 等）のcommit/pushもPMが実施

---

## 実行手順（全担当共通）

1. **inbox/ を確認**
   - 自分のディレクトリの `inbox/` フォルダ内のファイルを確認
   - ファイルがなければ「inbox/ に未処理のタスクはありません」と報告

2. **タスクを実行**
   - 指示ファイルの場合: 内容に従って作業を実行
   - 報告ファイルの場合: 内容を確認し、次のアクションを判断

3. **Git操作（コード変更がある場合）**
   - **自分の担当範囲のみ** を `git add` してcommit/push
   - 他担当のファイルは絶対にaddしない

4. **完了処理**
   - 処理したファイルを `archive/` に移動
   - `/pm/inbox/` に報告ファイルを作成

|-------------|--------|
| フロント | `/pm/inbox/` |
| バック | `/pm/inbox/` |
| インフラ | `/pm/inbox/` |

## ファイル命名規則

**指示ファイル:**
```
YYYY-MM-DD-NNN-task-title.md
例: 2026-01-21-001-deploy-investigation.md
```

**報告ファイル:**
```
YYYY-MM-DD-NNN-report-from-{担当名}.md
例: 2026-01-21-001-report-from-front.md
```

## 報告ファイルのフォーマット

```markdown
# 報告: [タイトル]

報告日: YYYY-MM-DD
報告者: [担当名]
対象タスク: [元の指示ファイル名]

## 実施内容
- [やったこと]

## 結果
- [結果・成果物]

## 課題・懸念
- [あれば記載]

## 次のアクション
- [必要なら記載]
```

## sharebox について

`/sharebox/` にはプロジェクト全体の指示・報告が記録されています。

- **管理者:** PMのみ
- **用途:** チャットクリア後の経緯復元、監査ログ
- **各担当の関わり:** 直接触らない（PMが自動で記録）

チャットをクリアして再開する場合は `/restart [担当]` を使ってください。
shareboxから経緯を復元してから作業を開始します。

---

## 注意事項

- 引数なしで `/GO` を実行した場合は、このヘルプを表示
- 不明な担当名が指定された場合は、有効な担当名（front/back/infra）を案内
- 外部サービスの操作が必要な場合は、必ず最新の公式ドキュメントを確認してから案内
- チャットが長くなったら `/clear` → `/restart [担当]` でリフレッシュ
